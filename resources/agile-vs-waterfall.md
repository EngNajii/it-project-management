# ⚖️ Agile vs. Waterfall – Quick Reference

Not sure which methodology fits your project? This guide helps you decide.

---

## At a Glance

| Factor | Waterfall | Agile |
|--------|-----------|-------|
| **Approach** | Linear, sequential | Iterative, incremental |
| **Planning** | Upfront, detailed | Continuous, adaptive |
| **Requirements** | Fixed at the start | Evolve throughout |
| **Delivery** | One final delivery | Frequent small releases |
| **Customer involvement** | At start and end | Continuous |
| **Change handling** | Difficult and costly | Expected and welcomed |
| **Documentation** | Heavy | Lightweight (just enough) |
| **Team structure** | Siloed by phase | Cross-functional |
| **Best for** | Stable, well-defined projects | Complex, evolving projects |

---

## When to Choose Waterfall

✅ Requirements are clear, complete, and unlikely to change
✅ Project has a fixed budget and hard deadline
✅ Regulatory or compliance requirements demand heavy documentation
✅ Team is geographically distributed or working across organizations
✅ Integration with third-party systems requires fixed contracts

**Common examples:** Government IT projects, infrastructure rollouts, ERP implementations with fixed scope

---

## When to Choose Agile

✅ Requirements are unclear or expected to evolve
✅ Speed to market is a priority
✅ Stakeholders want to see working software early and often
✅ Team is co-located or highly collaborative
✅ Innovation and exploration are part of the project

**Common examples:** Software product development, digital transformation, startup MVPs

---

## Hybrid Approach

Many real-world IT projects use a hybrid:
- **Waterfall for structure** – defined phases, milestone gates, budget control
- **Agile for execution** – sprints, backlog, continuous feedback within phases

> Example: Use Waterfall planning for a migration project (fixed dates, fixed scope) but run development in 2-week sprints.

---

## Popular Agile Frameworks

| Framework | Best For | Key Concept |
|-----------|---------|-------------|
| **Scrum** | Software dev teams | Sprints, Scrum Master, daily standups |
| **Kanban** | Ongoing work / support | Visual board, WIP limits, flow |
| **SAFe** | Large enterprises | Scaled Agile across multiple teams |
| **XP (Extreme Programming)** | Engineering-heavy teams | TDD, pair programming, continuous integration |

---

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| Sprint | A fixed time-box (1–4 weeks) for delivering work in Scrum |
| Backlog | Prioritized list of all work to be done |
| Velocity | How much work a team completes per sprint |
| WIP Limit | Maximum work-in-progress items allowed (Kanban) |
| MVP | Minimum Viable Product – smallest thing that delivers value |
| Epic | A large body of work broken into stories |
| User Story | A requirement written from the user's perspective |
| Definition of Done | Agreed criteria for when work is truly complete |

---

> 💡 **Rule of thumb:** The more uncertainty in your project, the more you benefit from Agile. The more fixed your constraints, the more structure (Waterfall) you may need.
